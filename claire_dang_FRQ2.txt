Part 3:
After seeing what the two system calls do, why do you think we had to add system calls for the
operations on condition variables? Why not just have these operations as functions in ulib.c as
we did for the spinlock? (4 pts)
    We need system calls for condition variables because cv_wait() and cv_signal() interact with kernel-managed process state and scheduling. cv_wait() must put a process to sleep, release the lock, and later wake up and reacquire the lock. cv_signal() must wake sleeping processes. Those require access to kernel data structures such as the process table and scheduler, which user-level code in ulib.c can't directly access. While the spinlock functions only perform atomic operations on shared memory so they can be implemented entirely in user space.
